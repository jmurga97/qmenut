# QMenut Backend Architecture

## Goals

QMenut backend is designed as a Cloudflare-first system with a small public surface and a
private application runtime behind it.

- Keep the web menu public for diners while protecting restaurant management, CRM, orders,
  payments, and internal operations.
- Use GraphQL for the application API, but keep the GraphQL runtime private and reachable only
  through a Cloudflare service binding.
- Put Clerk authentication and request authorization in a public BFF/Gateway Worker.
- Keep D1 as the source of truth, following the tenant and integrity rules defined in
  `apps/api/migrations/0001_initial_schema.sql`.
- Structure the Hono codebase using the same style as the RoncalPhoto API: app factory, central
  route registration, config/env parsing, shared error handling, and feature modules.

## Runtime Topology

```text
Browser / apps/web
        |
        | HTTPS
        v
Public BFF/Gateway Worker (Hono)
        |
        | Cloudflare service binding
        v
Private GraphQL Worker (Hono + GraphQL)
        |
        | D1 binding
        v
QMenut D1 database
```

### `apps/web`

`apps/web` is the public diner-facing React app. It can be opened by any client and should only
request data that is safe for anonymous users unless the current feature explicitly requires
authentication.

The public menu experience is scoped to branch-facing data: dishes, categories, menu metadata,
translations, tags, allergens, schedules, branch photos, and active promotions.

### Public BFF/Gateway Worker

The BFF/Gateway is the only backend Worker exposed to the internet. It owns:

- CORS and public HTTP routing.
- Clerk session verification.
- Anonymous versus authenticated request classification.
- Tenant resolution from host, custom domain, path, or Clerk organization context.
- Authorization decisions based on `restaurant_users`.
- Calls into the private GraphQL service through a Cloudflare service binding.

The public routes should start small:

- `GET /health`
- `POST /graphql`

### Private GraphQL Worker

The GraphQL Worker is the application runtime. It must not be publicly routed. It owns:

- GraphQL schema and resolver execution.
- Business services.
- D1 reads and writes.
- Transaction boundaries.
- Enforcement of tenant constraints passed by the BFF.

The private service route should be:

- `POST /graphql`, reachable only through the BFF service binding.

### D1 Database

D1 is the source of truth. The initial schema is intentionally strict about multi-tenant
integrity:

- Tables carrying both `restaurant_id` and `branch_id` use composite foreign keys back to
  `branches(id, restaurant_id)`.
- Public menu entities are branch-scoped.
- Soft deletes use `deleted_at`; public queries must exclude soft-deleted records.
- Money is stored as integer cents.
- Timestamps are integer epoch milliseconds.
- UUID values are generated by the Worker, not by D1.

## Request Flow

### Anonymous Menu Read

1. A diner opens a public menu route in `apps/web`.
2. The web app sends a GraphQL request to the public BFF.
3. The BFF resolves the target branch from the request host or route context.
4. The BFF creates an anonymous trusted context and calls the private GraphQL Worker through the
   service binding.
5. The GraphQL Worker executes only public menu resolvers for the resolved branch.
6. The response returns active, non-deleted, public-facing data.

### Authenticated Management Request

1. A restaurant user sends a GraphQL request to the public BFF.
2. The BFF validates the Clerk session.
3. The BFF resolves the Clerk user and organization into QMenut tenancy using
   `restaurants.clerk_org_id` and `restaurant_users.clerk_user_id`.
4. The BFF derives role and permissions from `restaurant_users`.
5. The BFF forwards a trusted context to the private GraphQL Worker.
6. The GraphQL Worker executes the requested resolver inside the tenant and permission boundaries.

## Security Model

The BFF is the security boundary. The GraphQL runtime trusts only requests received through the
Cloudflare service binding and must not be directly reachable from the public internet.

The GraphQL Worker should not validate Clerk directly during normal request handling. Clerk belongs
to the BFF layer so authentication, tenant lookup, and authorization remain centralized.

The BFF-to-GraphQL request context should include:

- Request id.
- Auth mode: `anonymous` or `clerk`.
- Clerk user id when authenticated.
- Clerk organization id when present.
- Resolved `restaurant_id` and `branch_id` when applicable.
- Role and permissions derived from `restaurant_users`.

Anonymous requests are allowed only for public menu reads. Any mutation or private query must require
an authenticated Clerk context and an authorized restaurant role.

## GraphQL Runtime

The private GraphQL Worker should be implemented with Hono and a GraphQL server adapter that works
inside Cloudflare Workers.

Resolvers should be grouped by domain modules rather than by transport concerns. A practical module
split is:

- `tenancy`: restaurants, branches, users, subscriptions, Stripe account metadata.
- `menu`: categories, dishes, variants, tags, allergens, translations, promotions, schedules, photos.
- `customers`: customers, customer restaurants, visits.
- `loyalty`: programs, rewards, transactions, redemptions.
- `orders`: delivery zones, orders, order items.
- `payments`: payment records and Stripe payment intent state.
- `campaigns`: campaigns and sends.
- `analytics`: daily menu and dish view rollups.
- `internal`: plan change logs and internal alerts.

Services should own business rules and DB access. Introduce repositories only when a module needs
multi-table transactions or repeated query composition. This keeps the code close to the RoncalPhoto
style without copying its Drizzle-specific implementation.

## BFF/Gateway

The BFF should stay thin. It should not duplicate GraphQL business logic.

Its responsibilities are:

- Parse and validate runtime environment.
- Configure Hono middleware.
- Apply CORS rules.
- Validate Clerk sessions.
- Resolve tenant and branch context.
- Convert request identity into the trusted internal context.
- Forward GraphQL requests to the private service binding.
- Normalize public error responses.

The BFF may inspect the GraphQL operation enough to enforce coarse access rules, but resolver-level
authorization should live with the GraphQL service because it has the schema, tenant context, and
domain knowledge.

## Public Web Data Contract

The anonymous web experience can read only active, branch-scoped, public-facing data.

Allowed public read areas:

- Branch identity: name, address, phone, WhatsApp, social links, custom domain, currency, schedules,
  and photos.
- Menu categories where `deleted_at IS NULL` and `is_active = 1`.
- Dishes where `deleted_at IS NULL` and `is_active = 1`.
- Dish availability windows.
- Variant groups and active variant options for public dishes.
- System and restaurant tags attached to public dishes.
- Allergens attached to public dishes.
- Translations for public menu entities.
- Active public promotions, including calculated effective prices from
  `v_dish_promotion_prices`.
- Delivery zones and ordering data only when the public ordering flow explicitly needs them.

Never expose these areas directly to anonymous users:

- `restaurant_users`.
- Subscription and Stripe account administration.
- Customer lists and loyalty balances outside an authenticated customer or restaurant flow.
- Campaign configuration and send history.
- Payment internals.
- Analytics rollups beyond intentionally public counters.
- Plan change logs and internal alerts.

## Data Model Overview

The current migration defines the backend around restaurant tenancy and branch-scoped menus.

### Tenancy

- `restaurants` maps QMenut restaurants to Clerk organizations through `clerk_org_id`.
- `branches` belong to restaurants and provide the branch/custom-domain boundary for public menus.
- `restaurant_users` maps Clerk users to restaurant roles: `owner`, `admin`, or `staff`.
- `branch_subscriptions` and `restaurant_stripe_accounts` support plan and Stripe state.
- `restaurant_languages` configures active and default languages per restaurant.

### Public Menu

- `categories` and `dishes` are restaurant and branch scoped.
- `dish_availability_windows` controls when dishes are visible or sellable.
- `dish_variant_groups` and `dish_variant_options` model configurable choices.
- `tags`, `dish_tags`, `allergens`, and `dish_allergens` describe dish metadata.
- `translations` stores translated fields for dishes, categories, variant groups, and options.
- `promotions`, `promotion_targets`, and `v_dish_promotion_prices` model public discounts and price
  calculation candidates.
- `branch_photos` and `branch_schedules` support the public branch page.

### User-Facing Flows

- `customers`, `customer_restaurants`, and `customer_visits` track diner relationships and visits.
- `loyalty_programs`, `loyalty_rewards`, `loyalty_transactions`, and `loyalty_redemptions` support
  points and stamps.
- `delivery_zones`, `orders`, and `order_items` support pickup and delivery ordering.
- `payments` stores Stripe payment intent status for orders.

### Business And Internal Flows

- `campaigns` and `campaign_sends` support Business-plan CRM campaigns.
- `menu_view_daily` and `dish_view_daily` store daily analytics rollups.
- `plan_change_logs` records plan transitions.
- `internal_alerts` tracks QMenut operational alerts.

## Suggested Source Layout

The backend can remain under `apps/api` while the split is being introduced, or later move into two
Worker packages if deployment needs it. In either case, keep the same internal shape.

```text
apps/api/src/
├── index.ts
├── app/
│   ├── create-app.ts
│   ├── routes.ts
│   └── middlewares/
├── config/
│   ├── env/
│   ├── handlers.ts
│   ├── status-codes.ts
│   └── types.ts
├── graphql/
│   ├── schema.ts
│   ├── context.ts
│   └── server.ts
├── modules/
│   ├── menu/
│   ├── tenancy/
│   ├── customers/
│   ├── loyalty/
│   ├── orders/
│   ├── payments/
│   ├── campaigns/
│   ├── analytics/
│   └── internal/
└── shared/
    ├── db/
    ├── errors/
    ├── lib/
    └── utils/
```

If the BFF and GraphQL runtime are split into separate apps, use the same pattern in both Workers:

- `create-app.ts` creates and configures the Hono app.
- `routes.ts` registers routes.
- `config/env` parses bindings and environment variables.
- `config/handlers.ts` owns not-found and error behavior.
- `shared/errors` owns common error types and mappers.
- `modules/*` owns domain behavior.

Do not copy RoncalPhoto's Drizzle or Better Auth setup. QMenut should use Clerk in the BFF, and the
database layer should use D1 prepared statements or a lightweight local DB access layer unless a
separate ORM decision is made later.

## Cloudflare Bindings

### BFF/Gateway Worker

Required bindings:

- GraphQL service binding, for example `GRAPHQL_SERVICE`.
- Clerk-related secrets and publishable configuration.
- Optional D1 binding if tenant lookup is performed directly by the BFF.

### Private GraphQL Worker

Required bindings:

- `DB`: D1 database binding for QMenut data.

Future bindings may include:

- Stripe secrets for checkout, subscriptions, and payment intents.
- Email Worker service binding for campaign and transactional email.
- Analytics Engine for raw event ingestion before daily rollups.
- R2 buckets for uploaded media.
- Queues for background jobs such as campaign sends, translation refreshes, purge jobs, and Stripe
  webhook processing.

## Operational Rules

- The GraphQL Worker must have no public route in production.
- Public menu queries must always scope by resolved branch and filter inactive or soft-deleted data.
- Authenticated operations must always scope by restaurant and, when relevant, branch.
- Clerk organization ids map to `restaurants.clerk_org_id`.
- Clerk user ids map to `restaurant_users.clerk_user_id`.
- Role checks should be explicit for owner/admin/staff operations.
- All D1 connections must enable foreign keys with `PRAGMA foreign_keys = ON`.
- Do not store effective promotional prices permanently; compute them from active promotions and
  `v_dish_promotion_prices`.
- Preserve the database convention of snake_case tables and columns.
- Keep generated files, build output, local Cloudflare state, and secrets out of source changes.

## Open Decisions

- Choose the GraphQL library/adapter for Cloudflare Workers.
- Decide whether BFF and GraphQL should live as two workspace apps immediately or start inside
  `apps/api` and split later.
- Decide whether the BFF performs tenant lookup directly through D1 or asks the private GraphQL
  service to resolve tenant context after Clerk verification.
- Define the exact permission matrix for `owner`, `admin`, and `staff`.
- Define how public customer/order flows authenticate customers, if customer accounts are required.
- Decide where Stripe webhooks terminate: BFF route, dedicated webhook Worker, or private service
  invoked by a public webhook receiver.
